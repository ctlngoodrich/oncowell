# Implementation Plan: OncoWell

> Generated by `app-build-planner`. Save as `docs/implementation-plan.md` in your project.

## Context

| Field | Value |
|-------|-------|
| App | OncoWell — a personal care navigation tool helping a patient understand treatment options (surgery vs. radiation), track health data, and prepare for doctor appointments |
| Need statement | Patients facing complex treatment decisions struggle to understand medical terminology, evaluate options in context, assess provider quality, and identify the right questions — leading to feeling overwhelmed and under-informed |
| Platform | Apple-native (Spezi Template Application) |
| Backend | None (on-device only with SwiftData). Architecture supports adding a backend later. |
| Study context | No |

## Planning Inputs

- **Need Statement** (`docs/planning/need-statement.md`): Defines the problem (treatment decision overwhelm), population (independent older adults with caregiver support), and outcome (confident, informed patient walking into appointments).
- **UX Brief** (`docs/planning/ux-brief.md`): Seven core journeys from onboarding through caregiver sharing. Trust is the central UX challenge — every piece of content shows its source. Minimal onboarding (3 steps max), no gamification, large text, VoiceOver support.
- **Data Model Brief** (`docs/planning/data-model-brief.md`): Nine core entities including Patient Profile, HealthKit Observations, Symptom Log, Doctor Visit Notes, Test Results, Appointments with recording/transcription, Question Lists, Treatment Content, and Glossary Terms. All on-device, FHIR-aligned.
- **Compliance Brief** (`docs/planning/compliance-brief.md`): Minimal compliance footprint. Key areas: recording consent reminders, transcription accuracy disclaimers, content freshness notices, and disclaimers that the app is not medical advice.
- **Study Brief**: Not applicable.
- **FHIR Data Model**: Not produced separately — FHIR recommendations are embedded in the data model brief.

## Milestones

### Milestone 1: Project Foundation

**Goal:** A running app with tab navigation, theming, and the basic screen structure — no functionality yet, but the skeleton is in place.

**Depends on:** Nothing

**Tasks:**
1. Rename the Spezi template project to "OncoWell" (bundle ID, display name, scheme)
2. Set up tab-based navigation with four tabs: Home, Learn, Track, Appointments
3. Create placeholder views for each tab
4. Configure app theming — large default text (16pt+ body), high contrast color palette, Dynamic Type support
5. Add the app icon placeholder and launch screen
6. Add general disclaimer text to an About/Info screen accessible from all tabs

**Platform notes:**
- Use SpeziViews for shared UI components (AsyncButton, ViewState, MarkdownView)
- Use Swift Charts framework for later health data visualization
- Configure the Xcode project for HealthKit and Speech Recognition entitlements (capabilities, not code yet)

**Verify:**
- App builds and runs on iPhone simulator
- Four tabs are visible and navigable
- Text respects Dynamic Type scaling
- About screen shows the general disclaimer

### Milestone 2: Onboarding

**Goal:** First-launch experience that welcomes the user, requests HealthKit access, and captures basic diagnosis context — all completable in under 60 seconds.

**Depends on:** Milestone 1

**Tasks:**
1. Build welcome screen with one-sentence app description and "Get Started" button
2. Build HealthKit permission screen explaining what data is accessed and why, with a "Skip" option
3. Build diagnosis context screen — cancer type (multiple choice), stage (simplified options), diagnosis date (approximate), treatment status — all skippable
4. Save onboarding completion state so it only shows once
5. Store diagnosis context in SwiftData as the Patient Profile entity

**Platform notes:**
- Use SpeziOnboarding (`OnboardingStack`, `OnboardingView`, `OnboardingActionsView`) for the step-by-step flow
- Use SpeziHealthKit for HealthKit authorization request
- Define the SwiftData `PatientProfile` model

**Verify:**
- First launch shows onboarding; subsequent launches skip it
- HealthKit permission dialog appears (or is skipped)
- Diagnosis context is saved and retrievable
- Entire flow completes in under 60 seconds

### Milestone 3: Treatment Content and Glossary

**Goal:** The user can browse surgery vs. radiation therapy side-by-side, read plain-language explanations with source citations, and tap any medical term for a definition.

**Depends on:** Milestone 1

**Tasks:**
1. Create the static content data structure — Treatment Content and Glossary Term models (can be Swift structs or bundled JSON)
2. Author initial treatment content for surgery vs. radiation: what it involves, recovery timeline, common side effects, success rates, quality-of-life considerations (sourced from NCCN/AMA/ACS)
3. Author glossary entries for key medical terms used in the content
4. Build the Learn tab — treatment comparison view with side-by-side or tabbed layout
5. Build individual treatment detail views with source citations displayed prominently
6. Build inline glossary — tap a highlighted term to see its plain-language definition
7. Add content disclaimer footer on all treatment content screens
8. Build provider evaluation guide content — what to look for, academic vs. community, checklist of questions

**Platform notes:**
- Use SpeziViews `MarkdownView` for rendering content if stored as markdown
- Content is static and bundled — no network requests
- Ensure all content meets 6th-grade reading level

**Verify:**
- Learn tab shows surgery and radiation options
- Each option has complete content with source citations (e.g., "Source: NCCN Guidelines for Patients, 2025")
- Tapping a medical term shows a plain-language definition
- Provider evaluation guide is accessible
- Content disclaimer is visible on every content screen

### Milestone 5: Symptom Logging and Health Tracking

**Goal:** The user can log symptoms daily and enter test results / lab values, all visible in a timeline on the Track tab.

**Depends on:** Milestone 1

**Tasks:**
1. Define SwiftData models: `SymptomLogEntry`, `TestResult`
2. Build symptom logging form — symptom type (picker), severity (1-5 or mild/moderate/severe), body location (optional), notes (optional)
3. Build test result entry form — test name (structured picker + free text), value, unit (auto-suggested), reference range (optional), date, notes
4. Build the Track tab — timeline view showing symptom logs, test results, and doctor visit notes in chronological order
5. Add delete capability for any user-created entry
6. Validate numeric inputs (e.g., PSA > 0, reasonable date ranges)

**Platform notes:**
- SwiftData for persistence
- Consider using SpeziQuestionnaire if symptom logging maps well to a FHIR Questionnaire, otherwise custom SwiftUI forms

**Verify:**
- Can log a symptom and see it in the timeline
- Can enter a test result with auto-suggested units
- Timeline shows entries in chronological order
- Entries can be deleted
- Input validation works (no negative values, no future dates for past events)

### Milestone 9: Transcript Intelligence

**Goal:** After transcription, the app generates a plain-language summary and extracts key topics (decisions, next steps, medications, follow-up instructions).

**Depends on:** Milestone 8

**Tasks:**
1. Implement summary generation using on-device intelligence (Apple Intelligence / Core ML)
2. Extract structured topics from transcript: decisions made, next steps, medications discussed, follow-up instructions
3. Map extracted questions back to the Question List (if linked) — mark which were answered
4. Display summary and extracted topics in a structured view within the appointment detail
5. Add ability for user to annotate/correct extracted topics
6. Ensure disclaimer is shown alongside AI-generated summaries

**Platform notes:**
- Apple Intelligence APIs (available on iPhone 15 Pro+ and later with iOS 18.1+)
- If Apple Intelligence is not available on the user's device, gracefully degrade — show transcript only, no summary
- This is the highest-effort custom feature. Consider a phased approach: transcript first (Milestone 8), intelligence later (this milestone)

**Verify:**
- Summary generates after transcription completes (on supported devices)
- Extracted topics are displayed in structured format
- User can annotate or correct extracted topics
- Graceful degradation on unsupported devices
- Disclaimer visible on all AI-generated content

## Open Questions

1. **Cancer type scope**: Should initial treatment content cover one cancer type (e.g., prostate) or multiple? Recommend starting with one and expanding — less content to author, more focused and relevant.
2. **Apple Intelligence availability**: Milestone 9 (Transcript Intelligence) requires Apple Intelligence, which needs iPhone 15 Pro or later with iOS 18.1+. What is your dad's iPhone model? If unsupported, Milestone 9 can be deferred or replaced with a simpler keyword extraction approach.
3. **Content authoring**: Treatment content and glossary entries need to be written or curated from NCCN/AMA/ACS sources. This is a content task, not a code task — it should happen in parallel with Milestone 1-2 development.
4. **HealthKit in simulator**: HealthKit data is limited in the iOS Simulator. For Milestone 4 testing, you may need a physical device or manually add sample data in the Health app.
5. **Lab result structure**: PSA and CBC are numeric, but pathology/biopsy results are narrative. Milestone 5 should support both but keep narrative results as simple free text.
6. **On-device vs. API intelligence**: Currently planned as on-device only. If summary quality is insufficient, adding an external LLM API (e.g., Claude) is possible but requires a compliance re-evaluation since data would leave the device.
7. **Question suggestion logic**: Milestone 7 suggests questions based on viewed content. The simplest approach: each Treatment Content item has a set of associated suggested questions. More sophisticated approaches can come later.

## Next Steps

This document is saved as `docs/implementation-plan.md` in the OncoWell project.

To start building, work through the milestones one at a time. Each milestone is designed to be built and reviewed as a single focused session. Milestones 1, 3, 5, and 7 can be worked on somewhat independently since they touch different tabs, but the dependency chain should be respected for the best experience.

Recommended starting order:
1. **Milestone 1** (foundation) — gets the app running
2. **Milestone 2** (onboarding) — first user experience
3. **Milestone 3** (content) — the core value proposition
4. **Milestone 4** (HealthKit) — brings in personal health data
5. Continue sequentially from there

Content authoring (treatment comparisons, glossary terms) should start now in parallel — it is the most time-consuming non-code task.
